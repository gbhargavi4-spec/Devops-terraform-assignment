# Security

## Secrets Management

No secrets are hardcoded anywhere in the codebase.

- **Database password** — generated by `random_password` at apply time, stored as a `SecureString` in SSM Parameter Store at `/<project>/<env>/rds/password`. The EC2 instance retrieves it at boot via the IAM instance profile.
- **GitHub Actions** — no long-lived AWS credentials. Authentication uses OIDC federation. A `aws-actions/configure-aws-credentials` step exchanges a GitHub-issued JWT for a short-lived STS token using `sts:AssumeRoleWithWebIdentity`.

## IAM — Least Privilege

| Role | Permissions |
|------|-------------|
| EC2 instance profile | CloudWatch PutMetricData (namespaced), SSM GetParameter for `/<project>/<env>/*`, ECR GetAuthorizationToken + pull, SSM Session Manager |
| GitHub Actions deploy (dev/qa/prod) | Targeted `terraform:*` via S3/DynamoDB state, ECR push to own repo, EC2 describe + SSM SendCommand |
| RDS Enhanced Monitoring | `rds:*` (AWS managed `AmazonRDSEnhancedMonitoringRole`) |

IAM policies use `aws_iam_policy_document` data sources — no inline JSON strings.

## Network Security

- RDS subnets have **no route table** — no path to/from the internet or NAT Gateway
- Security groups use the AWS Provider 5.x resource model (`aws_vpc_security_group_ingress_rule` / `aws_vpc_security_group_egress_rule`) — no inline rules, enabling independent lifecycle management
- EC2 IMDSv2 is **required** (`http_tokens = "required"`) — blocks SSRF-based metadata credential theft
- All EBS volumes and the RDS instance use **encrypted storage** (`storage_encrypted = true`)
- VPC Flow Logs capture all traffic for audit

## Container Security

- Multi-stage Docker build — final image is Alpine with no build tools
- Application runs as non-root user `appuser` (UID varies by Alpine)
- No secrets are baked into the image — they are injected via environment file at container start
- Trivy scans the image in CI on every push, blocking on CRITICAL and HIGH unfixed vulnerabilities

## GitHub Actions Hardening

- `permissions` blocks are set to minimum required on every workflow (`id-token: write`, `contents: read`)
- Production deployments require manual approval via the `production` GitHub Environment with required reviewers
- OIDC conditions restrict token acceptance to the specific `repo:org/repo:*` sub claim
